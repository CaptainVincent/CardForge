# 信用卡官網 → CardForge 規則 JSON

輸入一張信用卡的**官方網址**(或卡名),自由但有節制地探索官方頁面,把該卡的回饋規則
轉成 **可被 `importFromJson` 載入的 CardForge JSON**,並附上註記。產出前一定先讀 schema、
產出後一定用驗證腳本確認引擎真的吃得下。

## 黃金準則

- **先確認是哪張卡(identity-first)**:動手抓規則前,**先從給定 URL 確立卡片身分**(卡片正式
  名稱 + URL slug)。**絕不把臆測的卡名代入搜尋**——那會找到「你以為的卡」並自我確認(確認偏誤)。
  交付的卡**必須就是該 URL 指向的卡**;身分無法確認就問使用者,別猜。(實戰教訓:`C_uniopen`
  被誤當成 LINE Pay 卡,整個統一集團回饋抓錯。)
- **schema 先行**:動手前先完整讀 `references/schema.md`(三軸模型 + 全部構造 + 範例)。
  它是這份規則語言的權威;不要憑印象造欄位。
- **引擎是真相**:產出後**必須**執行 `scripts/validate.mjs` 並通過。它用 CardForge 真正的
  `importFromJson / nodeIssues / exportToJson / simulateMonth` 檢查,會抓出「看起來對但引擎
  不支援」的東西。沒過就修,修到過為止。
- **忠實 > 完整**:只寫官方頁面**明確寫出**的規則。推測、業界慣例、過期活動 → 寫進「註記」,
  不要混進規則。不確定的數字寧可標註待確認,也不要編。
- **別硬塞(no force-fit)**:一條規則**無法乾淨對映**到既有構造時,**不要走阻力最小的路**用
  最接近的構造近似。依序判斷:(a)**換個既有構造**能表達嗎?(擇一/取高/級距/計數級距/時段/
  門檻/上限…先窮舉再說);(b)是**常見且值得升級為一級原語**的型態嗎?→ 回報並升級(像 custom
  星期 → 一級 `day_of_week`);(c)**真的罕見**才用 `match.custom` 逃生口 + `note`。
  **明令禁止**:把「分階/計數型」壓成二元資格旗標、把「可用欄位表達的」塞進 `note`。
  (實戰教訓:踩點被壓成 on/off 旗標、互斥首刷被拆成兩旗標——都是 force-fit。)
- **回中文**:對話與註記用繁體中文。
- **錯了就回寫(self-improving)**:每次抓錯、被使用者指正、或被驗證/身分閘門擋下,**不要只修這一張卡**
  ——把根因與更正思維蒸餾成**一條耐久規則**,回寫進本 `SKILL.md`(放對應位置:身分類→黃金準則/步驟 2.0;
  映射類→「映射陷阱」;schema 類→更新 `references/schema.md`)。目標是讓**下一次的理解率提升**,讓 skill 隨
  每次實戰複利成長。詳見最後一節「出錯回顧 → 回寫 skill」。

## 流程

建議用 TodoWrite 建立以下步驟逐一完成。

### 1. 讀 schema
讀 `references/schema.md`。確認你掌握:`match`(含 exclude / or_groups / custom)、三種
`reward.method`、`tiers`(flat / spend / **marginal** / **distinct_count 計數級距**)、`limits.caps`(**reward / spend / count**
× txn / period / total,含共用 pool)、`eligibility`(門檻 `min_spending` + **資格 `flags`**)、
`eligibility_flags`(具名資格:新戶/登錄)、`select_group`(擇一:擇優/自選)、`top_group` + `top_groups`(取高)、
`point_programs`(點數估值)。

### 2. 有節制地探索官網(文字 → 圖片,兩層)

**2.0 先鎖定卡片身分(在抓任何規則之前;identity-first 的落實)**
- **解析 URL slug**:從給定網址取出卡片識別字(例 `…/C_uniopen.html` → `uniopen`;
  `…/LINEPay/…` → LINE Pay)。slug 是「這是哪張卡」的第一手線索,別忽略。
- 用 WebFetch 取頁,**先抓「卡片正式名稱」一項**確認身分,再抓規則。
- **頁面是 JS 空殼時(WebFetch 取到空白)**:搜尋一律**以「銀行 + slug 原字」為關鍵字**
  (例「中國信託 uniopen 信用卡 官網」),**不要**用你猜的產品名。目的只有一個:找到**同一張卡**
  的官方頁(優先 `mkt.*` / `minisite` / `/content/dam/` 等官方靜態頁;第三方部落格只能用來「定位
  官方頁」,不可當規則來源)。
- **身分核對閘門**:抓規則前,確認「找到的官方頁卡名」與「URL slug」一致。**不一致 → 停,重找或問
  使用者**。slug 與你即將寫的 `card` 名稱對不上,就是抓錯卡的警訊。

**2a. 文字優先(最可靠、最好解讀)**
- 從使用者給的 URL 開始,用 WebFetch 取得頁面。
- **範圍限制**:只在**同一官方網域**內,跟進與「權益 / 回饋 / 費率 / 注意事項 / 條款 / 活動辦法」
  直接相關的連結與 **PDF**(`Read` 可直接讀 PDF)。建議上限約 5–8 頁;避免無關連結、外部站、登入牆。
- 抓取重點:各通路/類別/特店回饋率、加碼條件、回饋上限(每月/每筆/總額/筆數)、最低消費門檻、
  排除項目、點數名稱與官方點值、輪動/自選類別機制、活動檔期與是否需登錄、海外手續費、新戶禮。

**2b. 讀圖(關鍵 — 因為 WebFetch 會丟棄圖片)**
信用卡頁的**指定特店/通路清單、費率圖**常常**只存在於圖片**,且銀行圖片多半沒有有用的 `alt`。
若文字層提到「指定特店 / 指定通路 / 最高 N%」卻沒有完整清單,**務必讀圖**,但要「猜對圖」:

```
node <skill 目錄>/scripts/find-card-images.mjs <頁面網址>
```
此工具依兩個已驗證的訊號(**鄰近回饋關鍵字** + **圖片尺寸**)把候選圖排序(大圖、近回饋文字者優先;
icon/CTA 按鈕自動沉底),並印出每張的文字脈絡。**對前幾張候選用 `Read` 視覺讀取**,抽出特店/費率。
- 不要逐張亂猜圖片網址(會讀到「申請按鈕」之類雜訊);用排序後的前幾張即可。
- 讀到的特店/費率務必與文字/條款**交叉驗證**;含糊的(例「其他指定通路」)以圖片清單為準。
- (腳本與本 `SKILL.md` 同目錄。)若該工具回報「0 張內容圖」或頁面為 **JS 動態渲染**(curl 抓到空殼),
  代表抓不到圖 —— 依 **2.0** 以「銀行 + slug」找**同一張卡**的官方靜態頁(`mkt.*`/`minisite`/`dam`),
  再用 WebFetch / `Read` 取內容;或請使用者直接提供該卡的內容/圖片網址。**切忌**因為原頁空白就改用
  臆測卡名搜尋而漂移到別張卡。

- 找不到關鍵數字就記下來,別猜。

### 3. 映射到構造(對照 schema)

**先做「機制覆蓋掃描」(逐卡一次,治『漏抓』)**:對照下列常見機制逐項自問「這張卡有沒有?」,
有就確認對映到哪個構造、沒有才略過。憑印象抓最容易漏不顯眼的(上限/排除/卡友日常被略過):

| 機制 | 構造 |
|---|---|
| 基本回饋(國內/海外/一般) | 多條 `match` + `reward` |
| 通路/類別/特店/支付 加碼 | `match`(channels/categories/merchants/payment_methods/mcc) |
| 消費滿額升級 / 超額累進 | `tiers` spend / marginal |
| 踩點等「計數→費率」 | `tiers` distinct_count(+count_label) |
| 自動取最高類別 / 自選類別 | `top_group` / 擇一·自選 |
| **卡友日 / 週幾 / 每月某號** | `match.day_of_week` / `day_of_month` |
| 需登錄 / 每月任務 | `eligibility.flags`(登錄) |
| 等級 / 身分 / 新戶vs既有(互斥) | 擇一·自選(`select_group` pick) |
| 滿額 / **滿筆** 解鎖 | `eligibility.min_spending`(metric spend / count) |
| 上限(每月/每筆/年/總額/筆數、共用池) | `limits.caps` |
| 首刷禮 / 新戶禮 | `period.from_opening_days` + `min_spending` + once |
| 點數 / 點值 | `reward.point_name` + `point_programs` |
| 排除項(不回饋/分期/稅費) | `match.exclude` 或 `note` |
| 多期 / 輪動 / 活動檔期 | dated rules + `period` |

逐條把規則落到正確的軸(**遇不乾淨對映 → 套黃金準則「別硬塞」**):
- 「滿額才開始享(%)」→ `eligibility.min_spending`(門檻);門檻是「解鎖後續消費的回饋」。
- 「**當月滿 N 筆**才享/才解鎖」→ `eligibility.min_spending` 的 `metric:"count"`(amount=筆數);
  與金額門檻同一套機制(可共用 pool)。別跟「前 N 筆才享(`caps metric:count`)」搞混——前者是
  **下限解鎖**(滿 N 筆後才開始),後者是**上限截斷**(只有前 N 筆有)。
- **「限新戶 / 需登錄活動才享」→ `eligibility.flags`(資格)**,卡層 `eligibility_flags` 列出名稱。
  **慣例:不寫 `default`(留未選),讓使用者自己選符合與否**——「我是不是新戶 / 有沒有登錄」是
  持卡人的事實,卡片不該替他預設。引擎把未選當「未符合」(日常試算不灌水);使用者在編輯器選、
  或在分析面板勾「符合」才亮起。**同一資格多條規則共用 → 寫同一個 `flags` 名稱**(一個開關控多條)。
  取代過去用 `is_active:false` 表達新戶、以及已退役的 `requires_activation`。
- **「當期累積滿額 → 送固定獎勵、給一次」**(如「當月滿 3,000 送 300」)= 門檻 + **固定金額**回饋,
  兩種等價表達,差在 300 計在哪:
  - **`settlement:"once"`**:列為一次性(放「另有一次性」清單、**不計入**當期回饋總額、繞過上限)。
    固定金額沒有上限問題,所以這是「一次性里程碑/滿額禮」最自然的表達。
  - **`{metric:"count", window:"period", max:1}`(不設 once)**:**計入**當期回饋總額。
  選哪個看你要不要把它算進「當月回饋總額」。
- **「只限某檔期」≠「結算方式」**:活動只在某段期間,用 `is_active`/`period` 表達(時間軸);
  「給一次」用上面的結算方式(發放軸)。兩軸獨立,別混。
- **唯一不能用 `once` 的:比例 + 上限累積型**(如「20% 上限 500」)——`once` 會繞過上限、只算
  第一筆 → 低估又爆上限。這種一律「每筆循環 + `window:"total"` 上限(+ 限時用 `is_active`)」。
- 「回饋上限 $X / 月」→ `caps metric:reward window:period`;「前 $X 消費享優惠」→ `metric:spend`;
  「前 N 筆」→ `metric:count`;「單筆上限」→ `window:txn`。
- 「超過 X 元的部分享 Y%」→ `tiers.mode:"marginal"`;「當期滿 X 元整筆升級」→ `tiers.mode:"spend"`。
- **「費率由當期某個『計數』決定」**(踩點:當期不同品牌數越多檔越高;或天數/筆數…)→
  `tiers.mode:"distinct_count"` + `count_label`(命名該計數,如「不同品牌數」)+ `bands[{min_count,rate}]`。
  通用機制(節點不綁特定用途);計數是**情境輸入**(分析給、記帳判定),引擎不自動推算。
  別硬塞成「資格旗標(有/沒有)」——那會失真成二元。
- **「卡友日/週幾/每月某號限定」**(週五夢時代 +8%、每月 1 號家樂福、週日康是美滿額送點…)→
  match 的 `day_of_week`(`["fri","sat","sun"]`)/ `day_of_month`(`[1,20]`):**交易日期屬性**,
  引擎由交易 `date` 自動推算星期/日,做成「一條 match 帶日期條件的靜態規則」。**不要**當動態情境、
  也不要用 `custom` 硬塞星期。買一送一/折價券/加購等贈品**非 %回饋** → 記 note,不結構化。
- 「數種優惠擇一」→ 同 `select_group`;「自動加碼最高消費類別」→ `top_group` + `top_groups{k}`。
- **官網活動檔期 → 填進該檔期所有規則的 `period.start`/`end`**(如「2026/1/1–6/30」→
  `start:"2026-01-01", end:"2026-06-30"`)。檔期內的加碼規則**每一條都要填**,別只填一條;
  卡片的常態基本回饋(非活動)可不填。⚠️ 引擎模擬**不依日期自動生效/失效**(日期屬記錄/
  匯出/編輯器顯示用);若要「過期就不算進試算」,改用 `is_active:false`。檔名後綴也依此檔期
  (上半年→`2026h1`)。
- **會員/身分等級制**(如 Level 1/2 上限或費率不同,取決於**帳戶身分**——大戶、資產餘額、
  方案等級——而非「這筆消費」):引擎無法從交易判斷身分,不可用單一規則表達「視身分而定」。
  **首選做法:擇一節點「自選(pick)」**——各等級一條規則(**相同命中、只差上限/費率**),
  都帶同一個 `stacking.select_group`,並在卡層 `select_groups:{<id>:{mode:"pick"}}`;成員以 `is_active`
  標出**預設採用哪級**(常見等級 `true`、其餘 `false`)。畫布上該擇一節點以 radio 切換、未選會警示。
  因命中相同,匯入會共用同一個條件/任一節點(不增節點)。範例見 `cards/sinopac-dual-currency-2026h1.json`。
  (替代:tier 自訂條件分流——適合「逐情境比較不同等級」;或 is_active 手動開關。)
- 點數 → `reward.point_name` + 頂層 `point_programs`(官方固定點值填 `basis:"fixed"` 與價格;
  彈性點/里程填 `basis:"estimate"`,價格可留給使用者維護)。
- 內建列舉沒有的條件 → `match.custom` 述詞。
- **個別細則/子限制 → 優先「結構化」,不要丟進 note。** 大規則底下的小備註(如「喬山限街邊
  門市」「Nuli 限官網訂閱」)其實多半能用元件表達:把那個特店**拆成自己的一條規則**,在它的
  condition 加上額外條件(例 `merchants:[喬山] + channels:[實體門市]`,或自訂 chip/述詞),
  再讓它與母規則**共用同一個上限池**(`limits.caps[].pool` + 卡層 `limit_pools`)。如此細則會被
  引擎真正執行(線上消費就不給加碼),而非只是文字。可結構化的判準:該限制是「某個可被
  match 欄位/自訂述詞描述的條件」。
  - **只有真的無欄位可表達的純文字**(如純說明、無法對應到任何交易屬性的限制)才寫進 `note`。

### 4. 產生 JSON(直接存進 `cards/`,依命名慣例)
- 用**資料庫形式** `{ "cards": [ ... ], "point_programs": {...} }`。
- 比率一律小數;每條規則 `tiers` 至少 `{mode:"flat"}`、`limits` 至少 `{}`。
- 規則 `name` 用人類可讀的中文摘要,`account` 用 `Liabilities:CreditCard:<Bank>:<Card>`。
- **存到 repo 的 `cards/` 資料夾**(不是 /tmp),檔名遵循 `cards/README.md` 的慣例:

  ```
  cards/<bank>-<card>-<period>.json
  ```
  - 全英文、小寫、連字號;用銀行**官方英文名**(永豐=sinopac、國泰=cathay、玉山=esun…),
    不確定時合理音譯。
  - `<period>` = 該卡回饋**適用週期**,由規則推斷:活動檔期上半年→`2026h1`、下半年→`2026h2`、
    全年活動→`2026`、全年一次→`annual`、月度常態→`monthly`、一次性→`once`。
  - 例:永豐 SPORT 卡(官網檔期 2026/1/1-6/30)→ `cards/sinopac-sport-2026h1.json`。
- ⚠️ **身分回核(identity-first 收尾)**:檔名 `<bank>-<card>` 與 `card` 名稱**必須呼應原始 URL 的
  slug**(例 URL 含 `uniopen` → 檔名 `ctbc-uniopen-…`、卡名「中國信託 uniopen 聯名卡」)。對不上 =
  你可能抓錯卡,**回步驟 2.0 重核**,別存檔。
- 產出後順手更新 `cards/README.md` 的「目前收錄」表格新增一列。

### 5. 驗證(必做,可迭代)
```
node <skill 目錄>/scripts/validate.mjs cards/<bank>-<card>-<period>.json
```
- 通過(✓)才算數。看 `features` 那行確認你想表達的構造(marginal / caps / 取高 / 擇一 / 門檻 /
  點數)數量符合預期 —— 若你以為加了某構造但 features 顯示 0,代表寫錯欄位,引擎沒認得。
- 未通過 → 依錯誤訊息修 JSON,重跑,直到通過。

### 5b. 交付前「逐條原文對照」自檢(必做 —— 提取錯誤多半在此攔下)

驗證只證明「引擎吃得下」,**不證明「忠於官網」**。交付前務必把產出的每條規則**逐一對回官網
原文**,逐項打勾(漏一項就是一個提取錯誤):

- [ ] **沒漏規則 / 沒多規則**:官網每一條加碼/基本回饋都有對應;沒有憑空多出來的。
- [ ] **回饋率**正確(小數;%、每元送點、固定金額分清)。
- [ ] **上限**:每月/每筆/每年/總額/**筆數** 都抓了?(上限與排除常**只寫在「注意事項/條款/
      活動辦法/FAQ」**,主視覺只有亮點 —— 沒讀條款頁就極可能漏上限。)
- [ ] **門檻**(滿額解鎖)vs **上限**(截斷)沒搞反。
- [ ] **資格**(限新戶/需登錄)只在官網明寫時才設;窗口(首刷 N 天)對。
- [ ] **檔期**日期對(收件/核准/活動起迄);過期不算進日常試算的用 `is_active:false`。
- [ ] **排除項**(不回饋交易、分期、稅費、儲值…)有結構化或記 note。
- [ ] **通路/特店清單完整**(圖片才有的清單有讀圖補齊,見 2b)。

- [ ] **計算基準**對:級距(spend/marginal)、計數(distinct_count + count_label)、取高(top)、
      擇一(select)、互斥身分→擇一·自選——別把 tiered 當二元、別把互斥當兩旗標(見映射陷阱)。

2. **一條通則**:把這次的「更正思維」抽象成一條可重用規則(不要只記這張卡的特例)。
3. **回寫位置**:

## 映射陷阱(務必避開 — 來自實戰修正)

**引擎是單期、只看「這筆消費」**:`period.start`/`end`(日期)會被 export/import 保留,但**模擬器不讀**
(無法依日期自動生效/失效)。`is_active`(編輯期停用)與 `eligibility.flags`(資格情境)**則會被
模擬器讀取**。所以:

- **限時 / 新戶取得型促銷**(如「核卡 45 天內綁 Pay 享 20% 刷卡金、限 N 名」)→ **用資格旗標
  `eligibility.flags:["新戶"]` + 卡層 `eligibility_flags:{"新戶":{"default":false}}`**:規則存著、
  編輯器看得到、`name`/`note` 標明「限時/名額」,但**預設不參與日常試算**(資格未符合→引擎跳過),
  不會把日常價值灌爆(例:讓 Apple Pay 看起來像 25%);使用者在分析面板勾「新戶✓」即可看到該情境。
  比舊的 `is_active:false` 更精確(語意是「需資格」而非「被停用」、可一個開關控多條;其下游與停用
  同樣顯示為無用路徑/變暗)。
  `window:"total"` 的上限仍照實填(代表整檔期上限)。
- **需登錄活動才享**(如 +1%/+3% 需先在 App 登錄)→ **`eligibility.flags:["已登錄"]` + 卡層
  `eligibility_flags:{"已登錄":{}}`**(不寫 default → 留未選;引擎當未符合、使用者在分析勾「已登錄✓」才算)。
  **只在官網明寫「需登錄/註冊活動」時才設**,不要推論;且某段登錄條件常只屬某幾條規則,
  別套到沒寫的(如新戶禮)。「綁定 Pay」屬 `match.payment_methods`,**不是**資格。退役提醒:不要再用
  `requires_activation`(已移除;舊檔匯入會自動轉成「已登錄」資格)。
- 一次性里程碑/首刷禮(完成一次給固定獎勵)才用 `settlement:"once"`;**百分比 + 上限的促銷不是
  一次性**(它隨消費累積),別誤用 once。
- `is_active:false` 現在只用於**純編輯期停用**(活動過期、暫時隱藏),以及 `select_group` 的 `pick`
  「擇一·自選」標出採用哪條;資格類請改用 `eligibility.flags`。
- **互斥的「一次性」回饋(新戶禮 vs 既有戶新卡 首刷禮、或方案二選一的里程碑)→ 用「擇一·自選」
  (`stacking.select_group` + 卡層 `select_groups:{<id>:{mode:"pick"}}`)把這些 `settlement:"once"`
  回饋串到同一個擇一節點,以 `is_active` 標出選哪張(預設皆 `false` = 未選);不要各掛一個獨立
  `eligibility.flags`。** 兩個獨立旗標可被同時打開 → 語意矛盾(會加總而非二選一)。引擎現有的
  `is_active` + once 處理天生給「恰好一張」,無需新構造。(實戰修正:uniopen 新戶/新卡禮原本誤用
  兩個資格旗標,應為一個擇一·自選。判準:互斥且「由持卡人身分決定」者 → 擇一·自選,不是資格旗標。)

**讓「指定特店」看得見,不要藏進 `or_groups`**:當某個加碼是針對**特定商家/通路**(7-11、App
Store…),用一級的 `match.merchants` 表達,讓它成為一條看得出來的規則。

- 若「指定支付」與「指定特店」**共用同一個上限**(例:兩者合計月上限 300),**拆成兩條規則 +
  共用上限池**(各自 `limits.caps[].pool` 指向同一個 pool,並在卡片層 `limit_pools` 註冊),而不是
  把它們塞進同一個 `任一(or_groups)` 規則 —— 後者會讓特店「藏起來」、使用者看不到。
- 只有當「同一條規則的命中條件本身是跨欄位的或」(且不需各自獨立顯示)時,才用 `or_groups`。

## 已知表達邊界(誠實告知使用者)

CardForge 目前**可表達**:多通路/類別比率、固定/每元送點、spend/marginal 級距、reward/spend/count
多重上限(單筆/每期/總額、可共用池)、消費門檻解鎖、**二元資格(新戶/登錄,具名共用、情境可切)**、
擇一(擇優/自選)、取高(動態最高消費類別)、排除、跨欄位任一、自訂述詞、點數估值。

**多期已支援**(交易帶 `date` 時):輪動季度 = 多條 dated 規則(各帶 `period.start/end`);各週期
上限/門檻依自己的 cycle(月/季/年/帳單週期,帳單週期需卡片 `statement_day`)自動重置;新戶首刷禮
(SUB)= `period.from_opening_days` + 卡片 `opened` + `min_spending.period:"total"` + `settlement:"once"`。

目前**尚未**精確表達(寫進註記,給近似):
- **階梯式上限**(上限本身隨消費級距變動)—— 用「每級距各一條規則 + 各自上限」近似。
- **轉點夥伴比例 / 兌換側**(到期、最低兌換)—— 模型專注 earn,兌換機制屬消費端。
